# Production Hardening (Pre-Phase-4) — Design

**Date:** 2026-04-16
**Goal:** Land 7 productization items so Phase 4 (PQC OCSP/CRL) ships into a system that can be operated in production without constant fire-fighting.
**Duration:** ~3 weeks of focused work.

## Context

The PQC CA system reached functional milestone today: cross-algorithm signing matrix complete, end-to-end leaf cert flow verified live on VPS through the RA portal. The cryptographic core is solid. What's missing is operational scaffolding — items that don't move the feature surface but separate "demo that works on Tuesday" from "system that survives Friday at 3am."

Phase 4 (PQC OCSP/CRL) is the next feature milestone. Before starting it, lock down the production foundations.

## Decisions made during brainstorming

- **Tenant isolation:** mirror the CA/RA per-tenant pattern; not a global keyed map. Isolation is a CA-grade requirement, not a property.
- **Secrets:** sops + age now for static config (DB password, signing salts, Erlang cookie). Vault deferred to BYOK milestone for dynamic secrets (HSM PINs, rotating DB credentials).
- **CI:** Gitea Actions on a self-hosted runner on the existing VPS.
- **Backup:** local hot snapshots (hourly, 24h retention) + remote cold archives via Restic (Hetzner Storage Box, daily/weekly/monthly).
- **Audit log completeness:** manual sweep + integration tests. No compile-time enforcement, no runtime middleware.

## Workstreams

### 1. Tenant isolation property test

**Why:** Schema-mode tenant isolation is the system's most load-bearing security guarantee. Today it's enforced by convention. A property test that asserts no cross-tenant SQL access is the cheapest, most durable way to catch regressions.

**Shape:**
- Single test file `test/integration/tenant_isolation_test.exs`.
- Setup creates two ephemeral tenants A and B with full schema-mode provisioning.
- Test attaches `:telemetry` handler on `[:ecto, :repo, :query]` events to capture SQL strings.
- Enumerates public engine APIs via `Module.__info__(:functions)` and exercises each as tenant A.
- Asserts no captured SQL contains tenant B's schema prefix.
- Skips functions whose argument types can't be auto-generated; those go on a manual review list.

**Success criteria:** test passes on current main; fails when a known-bad function (the existing `ceremony_watchdog` query) is enabled.

### 2. Health + readiness endpoints

**Why:** Load balancers and on-call automation need to distinguish "process up" from "process able to do work." The current single `/health` conflates them.

**Shape:**
- Each portal (CA, RA, Platform) and each engine (CA, RA, Validation, Platform) gets two endpoints.
- `GET /health/live`: returns 200 unconditionally; only 503 if the BEAM is dead (which means the request never gets there). Used by load balancer.
- `GET /health/ready`: returns 200 only if (a) DB pool can check out a connection within 100ms, (b) for engines with key stores: at least one tenant's keys loaded, (c) all critical GenServers respond to a `:ping` call within 100ms. Returns 503 with a JSON body listing failed checks otherwise.
- Engine `/health` rename → `/health/live`; existing callers updated.

### 3. Secrets via sops + age

**Why:** Today secrets are in `/opt/pki/.env` plain text on the VPS, generated by `deploy/generate-env.sh`. No version control, no rotation procedure, no audit. sops + age fixes all three with no new infrastructure to operate.

**Shape:**
- `deploy/install.sh` generates an age keypair on first install. Private key at `/etc/pki/age.key` (root, 0400). Public key committed to repo.
- Secrets file `deploy/secrets/production.env.sops` committed in encrypted form. Contains everything currently in `.env`.
- `deploy.sh` decrypts to `/opt/pki/.env` at deploy time, sets ownership and 0400.
- `deploy/SECRETS.md` documents: how to add a secret, how to rotate the age key, how to recover if the host's key is lost.
- Old `generate-env.sh` removed.

### 4. Audit log sweep + integration tests

**Why:** Customers and auditors will demand a complete, queryable audit trail. Today coverage is patchy — we noticed missing entries during the unlock-key flow this session.

**Shape:**
- Inventory: `docs/audit-coverage.md` lists every mutating endpoint (LiveView events, controller actions, engine API endpoints) with current audit status.
- Standardize fields: `actor_id`, `actor_username`, `tenant_id`, `target_type`, `target_id`, `action`, `details`, `ip_address`, `timestamp`.
- Add missing `audit_log` calls based on the inventory.
- `test/integration/audit_coverage_test.exs`: exercises one representative call per mutating endpoint, asserts an audit event with required fields was written. Runs in CI.

### 5. Gitea Actions CI

**Why:** Today the dev loop is "build on VPS, hope it works." Every regression we hit this session would have been caught by `mix compile --warnings-as-errors` in CI before the deploy.

**Shape:**
- Self-hosted runner on the same VPS, separate systemd unit `pki-ci-runner.service`, low-priority cgroup so it doesn't starve production.
- `.gitea/workflows/ci.yml` runs on every PR and push to main:
  - PostgreSQL service container.
  - `mix deps.get`, `mix compile --warnings-as-errors`, `mix test`, `mix credo --strict` (advisory).
- Branch protection on `main`: PR required, CI must pass.

### 6. Backup hot + cold

**Why:** A host crash today means starting from zero. Operator-error recovery (accidental drop, wrong UPDATE) takes manual `pg_dump` parsing.

**Shape:**
- **Hot:** `pg_dump --format=custom` of `pki_platform` every hour, encrypted with the same age public key as secrets, written to `/var/backups/pki/hot/`. Rolling 24-hour retention.
- **Cold:** Nightly `restic backup` to a Hetzner Storage Box (or equivalent S3-compatible). Restic repo encrypted with a password kept in sops secrets. Retention: daily ×30, weekly ×12, monthly ×12.
- Two systemd timers + two units. Both have `OnFailure=` triggering a notification script (initially: write to a file in `/var/log/pki/backup-failures.log`; later: ping a webhook).
- `deploy/RESTORE.md`: exact commands to restore from hot or cold, with the recovery time observed during the Week 3 drill.

### 7. One-click rollback

**Why:** Every production team eventually deploys a bad release. The recovery path needs to exist *before* it's needed at 3am, not improvised under pressure.

**Shape:**
- `deploy.sh` already deploys per-service tarballs to `/opt/pki/releases/<service>/`. Modify to:
  - Keep last 3 tarballs per service.
  - On install: symlink `<service>/current` → newly extracted directory, save previous as `<service>/previous`.
- New `deploy/rollback.sh <service>`:
  - Stop service.
  - Swap `current` ← `previous` (rename, preserving the now-previous as `previous-previous` until next deploy).
  - Restart service.
  - Curl `/health/ready` until 200 or 30s timeout.
  - Print outcome.
- Idempotent; safe to run multiple times.

### 8. Final integration drill

Not new code — exercising the system. Pretend a host-fatal incident. Restore from hot backup. Verify every tenant's key ceremony state is intact, every CA can sign a fresh CSR, every RA can submit and approve. Time it. Update `RESTORE.md` with reality.

## Out of scope

- Multi-region HA, K8s, auto-scaling.
- Compliance program (SOC 2, WebTrust).
- Vault for dynamic secrets — comes with BYOK milestone.
- Metrics endpoint (Prometheus) — useful but not blocking; can land alongside Phase 4.
- Rate limiting on RA API — same.
- SSO/SCIM.

## Success criteria

After all 8 workstreams land:
- Tenant isolation property test passes in CI on every PR.
- Both `/health/live` and `/health/ready` exist on every service; load balancer config can distinguish them.
- Zero secrets in plaintext on the VPS outside `/opt/pki/.env` (which is 0400 root) or in git.
- 100% of mutating endpoints write audit events with the standard field set.
- CI runs on every PR; merging `main` requires green CI.
- Hourly hot backup runs; nightly cold backup runs and has been verified by restore.
- `deploy/rollback.sh` swaps back to the previous release in under 30 seconds.
- Recovery drill completed with documented timing.

Then: write the Phase 4 plan.
